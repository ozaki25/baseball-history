# 要件定義書 — 観戦履歴アプリ（再構築）

> ステータス: **ドラフト（レビュー中）** / 最終更新: 2026-07-10
> 本書は既存 `PROJECT_REQUIREMENTS.md` / `FUNCTIONAL_DESIGN.md` / `DETAILED_DESIGN.md` を置き換える前提のドラフトです。

## 1. プロジェクト概要

### 1.1 目的

北海道日本ハムファイターズの**個人の観戦記録**を管理・閲覧するアプリを、要件から整理し直して再構築する。
既存版は「毎ビルド時に公式サイトをスクレイピングして表示」する構成で、以下の課題があった。

- 公式サイトの応答・構造に**ビルドが丸ごと依存**（落ちる・重い・構造変更で即失敗）
- 過去の確定データを**毎回取り直す**無駄
- 表示とデータ取得が密結合で、拡張・保守がしにくい

再構築では **「取得（スクレイピング）と表示を分離し、確定データを永続化する」** ことを軸に据える。

### 1.2 ゴール / 非ゴール

| | 内容 |
|---|---|
| ゴール | 観戦記録を**多条件で絞り込み・集計**して振り返れる、軽量で壊れにくい個人用アプリ |
| ゴール | スクレイピングをビルド経路から切り離し、`games.json` を単一の信頼できるデータ源にする |
| ゴール | 現状の PWA・アクセシビリティ・ファイターズカラーの体験を踏襲する |
| 非ゴール | 複数ユーザー・認証・共同編集（完全に個人用） |
| 非ゴール | グラフ・チャートによる可視化（今回は数値集計と絞り込みが主役） |
| 非ゴール | リアルタイム更新・試合速報 |

### 1.3 利用者・利用シーン

- 利用者: 本人のみ（1 名）。認証は設けない（または将来的な最小限のみ）。
- 想定シーン: 観戦後に記録を確認、シーズン中/オフに「どの球場・相手に強い/弱い」「今年は何勝何敗」などを振り返る。
- 想定端末: **スマートフォンがメイン**（観戦先・移動中での閲覧/振り返り）。PC は補助。
  → UI は**モバイルファースト**で設計し、PC は広い画面を活かす拡張として扱う。

## 2. 用語

| 用語 | 定義 |
|---|---|
| 観戦記録（Game） | 本人が現地観戦した 1 試合分のデータ |
| 観戦日（dates） | 年ごとの観戦日リスト（`MMDD`）。取り込みの入力 |
| 取り込み（ingest） | 観戦日をもとに公式サイトを取得・解析し `games.json` を生成する処理 |
| 手動補完（override） | 取り込みで取れない/誤るデータを手動で上書きする仕組み |
| ホーム/ビジター | ファイターズ主催試合か否か |

## 3. 機能要件

### 3.1 一覧表示（FR-01）

- 全観戦記録を一覧表示する（PC: テーブル / モバイル: カード）。
- 表示項目: 日付・対戦相手・ホーム/ビジター・結果（勝/負/分/中止）・スコア・球場・メモ。
- 並べ替え: 日付・対戦相手・球場・結果など主要列で昇順/降順。

### 3.2 絞り込み（FR-02）

以下の **5 軸**で絞り込める。複数軸の AND 併用が可能。

| 軸 | 種類 | 例 |
|---|---|---|
| 年度・期間 | 単一/範囲 | 2025 年、2015〜2016 |
| 球場 | 複数選択 | エスコンフィールド、東京ドーム |
| 対戦相手 | 複数選択 | オリックス、ソフトバンク |
| ホーム/ビジター | トグル | ホームのみ |
| 勝敗 | 複数選択 | 勝ち／サヨナラ勝ちのみ |

- **絞り込み条件は URL に保持**する（ブックマーク・共有・ブラウザ戻るで復元可能）。
- 条件クリア（リセット）操作を用意する。

### 3.3 集計・統計（FR-03）

- **絞り込み結果に連動**した集計サマリを表示する。
- 基本統計: 観戦数・勝・負・分・（中止）・勝率。
- 軸別クロス集計（絞り込みと連動）:
  - 球場別の観戦数・勝敗・勝率
  - 対戦相手別の観戦数・勝敗・勝率
  - 年度別の観戦数・勝敗・勝率
  - ホーム/ビジター別の勝敗・勝率
- ※ グラフは描かない。数値テーブル・サマリカードで表現する。
- 記録が無い年度（例: 2017–2019, 2023）も年度別表示に含め、**「データなし（0件）」と明示**する（隠さない）。
- **対象範囲は上記の基本統計＋軸別集計まで**。連勝/連敗・通算得失点・サヨナラ勝ち数などの「実績」系指標は今回**作らない**。

### 3.4 データ取り込み（FR-04）

- `data/dates.json`（観戦日）を入力に、公式サイトを取得・解析して `data/games.json` を生成/更新するスクリプトを提供する。
- **データ源はスクレイピングのみ**。手で編集するルートは設けない（`games.json` は取り込みの生成物として扱う）。
- **アプリ本体はスクレイピングを一切行わない**（`games.json` を読むだけ）。
- 再実行は**冪等**（同じ入力から同じ結果）。既定は未取得分のみ取得、`--force` で再取得。
- 取得失敗・解析失敗の試合は `ingest-report.json` に記録し、再実行でリトライする。
- **取り込みの実行は GitHub Actions**（ビルド時でもアプリ実行時でもない）。`dates.json` の push または手動実行で起動し、生成した `games.json` をリポジトリへ自動コミット → Vercel が自動デプロイ。利用者は**観戦日を追加するだけ**（定期実行はしない）。

### 3.5 PWA（FR-05）

- ブラウザからインストール可能（Web App Manifest）。
- オフラインでアプリシェルと `games.json` を閲覧可能（Service Worker）。

## 4. 非機能要件

| 分類 | 要件 |
|---|---|
| パフォーマンス | データ数百件規模を想定。クライアント側集計で十分。初期表示は軽量に保つ |
| 可用性 | 表示は外部サイトに非依存（静的データ）。公式サイト停止時も閲覧可 |
| 保守性 | 取得・解析・表示・集計を疎結合に。型安全（TypeScript strict） |
| テスト容易性 | パーサ・集計ロジックを純関数化し単体テスト可能に |
| アクセシビリティ | WCAG 2.1 AA 踏襲（コントラスト・キーボード操作・ARIA・セマンティック HTML） |
| 対応環境 | **モバイルファースト**。iOS Safari / Android Chrome を最優先で最適化。PC 版（Chrome/Safari/Edge/Firefox 最新）は拡張対応 |
| 操作性 | タッチ操作前提（十分なタップ領域、片手操作を考慮、スクロール量を抑える） |
| 言語 | 日本語 UI のみ |
| デザイン | ファイターズ公式カラー踏襲（プライマリ `#016298` 等） |
| アイコン | アプリ/ファビコンを刷新。**完全オリジナル**（公式ロゴ・「F」マーク・ワードマーク・マスコットは使わない）で、チームカラー＋一般的な野球意匠により“らしさ”を表現。作成はフェーズ5 |
| デプロイ | Vercel（静的配信） |

## 5. データ要件（概要）

詳細は `design.md` を参照。中核となる `games.json` の 1 レコードは以下を最低限持つ。

- `id`（`YYYY-MM-DD` ベースの一意キー）
- `date`（`YYYY-MM-DD`）
- `opponent`（対戦相手・正規化名）
- `stadium`（球場・正規化名）
- `homeAway`（`home` / `away`）
- `result`（`win` / `lose` / `draw` / `cancelled`）
- `score`（`{ fighters, opponent }`）

> 手入力ルートを持たないため、`note`（観戦メモ）等の**スクレイピングで取得できない項目は持たない**。

## 6. スコープ

### 6.1 対象
- 一覧・絞り込み・集計の閲覧アプリ
- 取り込みスクリプトと GitHub Actions による自動更新
- PWA・Vercel デプロイ
- 旧 Next.js 資産の撤去と置き換え

### 6.2 対象外（今回やらない）
- 認証・複数ユーザー・共同編集
- グラフ/チャート可視化
- データの手入力・手動編集（データ源はスクレイピングのみ。編集 UI も override ファイルも設けない）
- ダブルヘッダー第2試合（レアかつ両方観戦しないため。URL は末尾 `01` 固定＝第1試合のみ扱う）
- 試合速報・通知

## 7. 確定した論点（レビュー済み）

1. **`games.json` の項目**: MVP は中核項目（§5）のみ。スクレイピングで取れる追加項目は、実データを見て必要なら後追いで検討。手入力項目は対象外。
2. **統計項目の範囲**: 基本統計＋軸別集計まで（FR-03）。実績系（連勝/連敗・得失点・サヨナラ数）は**作らない**。
3. **正規化の範囲**: 最小限（trim・全半角）を標準実装。チーム/球場の名寄せは**実データの distinct 値を確認してから**、割れていた分だけ後追いで対応（データ駆動）。
4. **欠損年（2017–2019, 2023 等）**: 隠さず「データなし（0件）」として表示（FR-03）。
5. **取り込みの実行**: GitHub Actions（`dates.json` push＋手動実行、cron なし）。詳細は `design.md` / `development.md`。
