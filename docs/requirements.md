# 要件定義 — 観戦ノート

北海道日本ハムファイターズの**個人観戦記録**を、多条件で絞り込み・集計して振り返る PWA。

- 対応する設計は [`design.md`](design.md)、開発運用は [`development.md`](development.md)。
- 依存バージョンは [`package.json`](../package.json) を単一の定義元とする（本書では版番を持たない）。

## 1. 目的とスコープ

### 1.1 目的

観戦記録の**閲覧・集計・振り返り**を、軽量で壊れにくく行う。既存版はビルド時に公式サイトを毎回スクレイピングしていたため
「サイトが落ちるとビルドも落ちる／確定データを毎回取り直す／表示と取得が密結合」という課題を抱えていた。
本アプリでは **「取り込み(ingest)」と「表示(app)」を完全に分離し、確定データ (`data/games.json`) を永続化する** ことを軸とする。

### 1.2 ゴール / 非ゴール

| 区分     | 内容                                                                            |
| -------- | ------------------------------------------------------------------------------- |
| ゴール   | 観戦記録を多条件で絞り込み・集計して振り返れる、軽量で壊れにくい個人アプリ      |
| ゴール   | スクレイピングをビルド経路から切り離し、`games.json` を単一の信頼データ源にする |
| ゴール   | PWA・アクセシビリティ・ファイターズカラーの体験を踏襲する                       |
| 非ゴール | 複数ユーザー・認証・共同編集                                                    |
| 非ゴール | グラフ・チャート可視化                                                          |
| 非ゴール | リアルタイム更新・試合速報                                                      |
| 非ゴール | データの手入力・手動編集（データ源はスクレイピングのみ・編集 UI も持たない）    |
| 非ゴール | ダブルヘッダー第2試合の扱い（第1試合のみ・URL 末尾 `01` 固定）                  |

### 1.3 利用者・利用シーン

- 利用者は本人のみ（1 名）。認証は設けない。
- 想定シーン: 観戦後の記録確認、シーズン中/オフの振り返り（球場別・相手別・年別など）。
- 想定端末: **スマートフォンがメイン**。PC は補助。UI は**モバイルファースト**。

## 2. 用語

| 用語                  | 定義                                                                             |
| --------------------- | -------------------------------------------------------------------------------- |
| 観戦記録（Game）      | 本人が現地観戦した 1 試合分のデータ                                              |
| 観戦日（dates）       | 年ごとの観戦日リスト（`MMDD`）。取り込みの入力                                   |
| 取り込み（ingest）    | 観戦日をもとに公式サイトを取得・解析し `games.json` を生成する処理               |
| 主催 / ビジター       | ファイターズ主催試合か否か（`homeAway: "home"` / `"away"`）                      |
| 観戦予定（scheduled） | 事前登録済みで結果未確定。結果が出た実行で自動的に確定へ更新される               |
| 詳細不明（unknown）   | 現行サイトで取得不能な古い試合。`data/date-only.json` に日付を挙げて日付のみ残す |

> 「詳細不明(unknown)」は値の**上書き(override)ではなく「詳細を持たないことの宣言」**として保存する。
> 集計では観戦数に含めるが勝敗軸には数えず、年度別軸には観戦数として現れる（§3.3）。

## 3. 機能要件

### 3.1 一覧表示（FR-01）

- 全観戦記録を一覧表示する（PC: テーブル / モバイル: カード）。
- 表示項目: 日付・対戦相手・主催/ビジター・結果（勝/負/分/中止）・スコア・球場。
  スクレイピングで取得できない項目（メモ等）は持たない（§5）。
- 並べ替え: 日付・対戦相手・球場・結果など主要列で昇順/降順。

### 3.2 絞り込み（FR-02）

以下の **5 軸**で絞り込める（複数軸は AND）。

| 軸            | 種類      | 例                             |
| ------------- | --------- | ------------------------------ |
| 年度・期間    | 単一/範囲 | 2025 年、2015〜2016            |
| 球場          | 複数選択  | エスコンフィールド、東京ドーム |
| 対戦相手      | 複数選択  | オリックス、ソフトバンク       |
| 主催/ビジター | トグル    | 主催のみ                       |
| 勝敗          | 複数選択  | 勝・負・分・中止               |

- **絞り込み条件は URL に保持**する（ブックマーク・共有・戻る/進むで復元）。
- 条件クリア（リセット）操作を用意する。

### 3.3 集計・統計（FR-03）

絞り込み結果に**連動**した集計サマリを表示する。基本統計 + 軸別クロス集計まで。
連勝/連敗・通算得失点・サヨナラ勝ち数などの「実績」系は作らない。

**基本統計** — 観戦数・勝・負・分・中止・勝率。

**集計の定義（野球の通例）**:

| 値                      | 観戦数(`attended`) | 勝敗 (`win`/`lose`/`draw`) | 勝率の分母 (`勝 + 敗`) |
| ----------------------- | :----------------: | :------------------------: | :--------------------: |
| `win` / `lose` / `draw` |         ✓          |             ✓              |     勝・敗のみ含む     |
| `cancelled`（中止）     |         ✓          |             ✗              |           ✗            |
| `unknown`（詳細不明）   |         ✓          |             ✗              |           ✗            |
| `scheduled`（予定）     |         ✗          |             ✗              |           ✗            |

**軸別クロス集計**（絞り込み連動）: 球場別 / 対戦相手別 / 年度別 / 主催-ビジター別の観戦数・勝敗・勝率。

- 記録の無い年度も年度別表示に含め **「データなし（0件）」** と明示する（隠さない）。
- `unknown` は相手/球場/主催の3軸では値が空のため自然に除外され、**年度別軸には観戦数として現れる**
  （`date` を持つため年度に振り分けられる）。
- グラフは描かない。数値テーブル・サマリカードで表現する。

### 3.4 データ取り込み（FR-04）

- `data/dates.json`（観戦日）を入力に、公式サイトを取得・解析して `data/games.json` を生成する。
- **アプリ本体はスクレイピングを一切行わない**（`games.json` を読むだけ）。
- 手編集ルートは設けない（`games.json` は取り込みの生成物として扱う）。
- 再実行は**冪等**（同じ入力から同じ結果）。既定は未確定分のみ取得、`--force` で全件再取得。
- 試合前の日は `scheduled`（観戦予定）として保存し、後日結果が出た実行で自動確定する。
- 取得/解析失敗は `ingest-report.json` に記録し、再実行でリトライする。
- **実行は GitHub Actions** で完結する（開発環境 / Vercel ビルドからは外部サイトへ到達できなくてよい）。
  `dates.json` の push または手動実行で起動し、生成した `games.json` をリポジトリへ自動コミット → Vercel が自動デプロイ。

### 3.5 観戦日の登録（FR-05）

- 観戦日の追加は **Claude 経由の専用コマンド** [`add-date`](../.claude/commands/add-date.md) で行う。
- コマンドは `dates.json` へ追記・重複スキップ・ソートし、commit & push まで実施（→ 取り込みが自動起動）。
- 事前登録（まだ試合前の日）にも対応する。

### 3.6 PWA（FR-06）

- Web App Manifest によりインストール可能。
- Service Worker によりオフラインでアプリシェルと `games.json` を閲覧可能。

## 4. 非機能要件

| 分類             | 要件                                                                                                                    |
| ---------------- | ----------------------------------------------------------------------------------------------------------------------- |
| パフォーマンス   | データ数百件規模を想定。クライアント側集計で十分。初期表示は軽量に保つ                                                  |
| 可用性           | 表示は外部サイトに非依存（静的データ）。公式サイト停止時も閲覧可                                                        |
| 保守性           | 取得・解析・表示・集計を疎結合に。TypeScript strict                                                                     |
| テスト容易性     | パーサ・集計ロジックを純関数化し単体テスト可能に                                                                        |
| アクセシビリティ | WCAG 2.1 AA 踏襲（コントラスト・キーボード操作・ARIA・セマンティック HTML）                                             |
| 対応環境         | モバイルファースト。iOS Safari / Android Chrome を最優先。PC 版（Chrome/Safari/Edge/Firefox 最新）は拡張対応            |
| 操作性           | タッチ操作前提（十分なタップ領域、片手操作を考慮、スクロール量を抑える）                                                |
| 言語             | 日本語 UI のみ                                                                                                          |
| デザイン         | ファイターズ公式カラー踏襲（プライマリ `#016298` 等）                                                                   |
| UI 独自性        | AI 生成感のある見た目は禁止。スコアブック由来の独自の世界観で作る（`design.md` §6.4）                                   |
| アイコン         | 完全オリジナル（公式ロゴ・「F」マーク・ワードマーク・マスコットは使わない）で、チームカラー＋一般的な野球意匠により表現 |
| デプロイ         | Vercel（静的配信）                                                                                                      |

## 5. データ要件（概要）

中核となる `games.json` の 1 レコードは次を最低限持つ（詳細スキーマは [`design.md` §4](design.md#4-データモデル) が唯一の定義元）。

- `id` — `YYYY-MM-DD` ベースの一意キー
- `date` — ISO `YYYY-MM-DD`
- `opponent` / `opponentId` — 対戦相手（当時の表示名 / 安定ID）
- `stadium` / `stadiumId` — 球場（当時の表示名 / 安定ID）
- `homeAway` — `home` / `away` / `null`（中止・予定・詳細不明で不定時）
- `result` — `win` / `lose` / `draw` / `cancelled` / `scheduled` / `unknown`
- `score` — `{ fighters, opponent }`（中止・予定・詳細不明時は `null`）

> 集計・絞り込み・URL は**安定 ID**（`opponentId` / `stadiumId`）で束ね、表示は**当時の名称**を用いる（球団改称・球場命名権対応）。
> 手入力ルートを持たないため、`note`（観戦メモ）等の**スクレイピングで取得できない項目は持たない**。

**追加項目の扱い**: MVP は上記の中核項目のみ。**スクレイピングで取れる追加項目**（例: 観客数・
先発投手など）は必要になった時点で、実データを見て後追いで検討する（データ駆動）。
手入力項目は対象外。

## 6. 設計上の確定事項

技術的な選定・具体構造は `design.md` / `development.md` を参照。要件レイヤで確定している事項は以下。

- **データ源はスクレイピングのみ**。手入力・値の上書きは行わない。
- **完全静的（SSG / prerender）** で配信する（アプリ実行時にスクレイピングしない）。
- **取り込みは GitHub Actions で実行**（cron なし・`dates.json` push か手動実行がトリガー）。
- **依存バージョン方針**: 実装着手時に npm レジストリで最新を確認して採用する（知識ベースの古い選定をしない）。
  採用済み版は `package.json` を単一定義元とする。
